Developer

The basic philosophy of Molecule is to think from the User’s perspective towards a technical solution. The priority is always to make it as easy as possible for the end User, no matter how challenging on the backend.

When we asked

“how can I most intuitively and with most minimal code query for persons?”

we first tried something like this

import Person._
m(firstName ~ lastName ~ age) // all fields of a `Person` trait...

But it quickly became not so intuitive with relations, expressions etc.

Builder pattern

After some different experiments we settled on using the builder pattern

Person.firstName.lastName.age

It seems to be the most minimal representation of what we want to express - in human terms, and with as little technical noise as possible.

Generate boilerplate code

The builder pattern has shown a surprising strong capacity to express a wide range of query constructs. But it also requires an extensive amount of boilerplace code to work.

We therefore generate all boilerplate code from a minimal Schema definition file automatically when we compile our project with sbt compile.

Each attribute is for instance defined as a field of a namespace trait stating what type and cardinality and possible options:

val name = oneString.fullTextSearch // maybe more options...

From this we generate 2 artifacts:

  1. Schema transaction data (in a Datomic format)
  2. Scala code to build molecules

As an example: to find

Names of twitter/facebook_page communities in neighborhoods of southern districts

we can compose a “molecule query” that is very close to our human sentence:

Community.name.`type`("twitter" or "facebook_page")
  .Neighborhood.District.region("sw" or "s" or "se")

Molecule transforms this at compile time (with macros) to a little more elaborate Datalog query string and input rules that finds those communities in the Datomic database:

[:find ?a
 :in $ %
 :where
   [?ent :community/name ?a]
   (rule1 ?ent)
   [?ent :community/neighborhood ?c]
   [?c :neighborhood/district ?d]
   (rule2 ?d)]

INPUTS:
List(
  datomic.db.Db@xxx,
  [[[rule1 ?ent] [?ent :community/type ":community.type/twitter"]]
   [[rule1 ?ent] [?ent :community/type ":community.type/facebook_page"]]
   [[rule2 ?d] [?d :district/region ":district.region/sw"]]
   [[rule2 ?d] [?d :district/region ":district.region/s"]]
   [[rule2 ?d] [?d :district/region ":district.region/se"]]]
)

Benefits

By not having to write such complex Datalog queries and rules “by hand”, Molecule allows you to

  • Type less
  • Make type safe queries with inferred return types
  • Use your domain terms directly as query building blocks
  • Model complex queries intuitively (easier to understand and maintain)
  • Reduce syntactic noise
  • Focus more on your domain and less on queries

Possible drawbacks

We still need to explore how far Molecule can match the expressive powers of Datalog. So far, all examples in the [Seattle tutorial][seattle] have been “molecularized” succesfully (see the [Molecule Seattle tutorial][tutorial] and [code][tutorialcode]). So as a proof-of-concept it looks promising…
